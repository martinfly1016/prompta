## Part A — 日报数据异常的 3 层根因排查

### 1. personal-color paywall_view 19 次 vs DB 仅 1 anon exhausted

**Root cause**: Phase 0 (5/11 `df57544`) 把 `/api/tools/personal-color/check` 的 `canUse` 重定义为：
```ts
const canUse = !!eh && balance > 0
const blockReason = !eh ? 'login_required' : balance > 0 ? 'none' : 'credits_exhausted'
```

但 `PersonalColorQuotaGate.tsx:295` 的 `handlePickFile` 还是老语义：
```ts
if (!state.canUse) {
  openPaywall(setShowModal, 'exhausted_pick')   // ← 不分场景
}
```

→ **未登录用户点「写真を選択」按钮** = 触发 `paywall_view.exhausted_pick`，但他们一次 free 都没用过。19 次里 18 次是这种「未登录路过」用户，**1 次是真撞墙**（5/9 04:58-05:00 anon `5e225236-edf2` 3 次 free in 1 分 22 秒）。

trigger 名 "exhausted_pick" 的语义在 Phase 0 后已死。**真实数据上「拉新阻力」和「复购阻力」混在一起**，view→click 4.8% 不可执行。

**Fix (`9c03f8b`)**: PaywallTrigger 加 `'login_required'`，按 `state.blockReason` 分流：
```ts
const trigger: PaywallTrigger =
  state.blockReason === 'login_required' ? 'login_required' : 'exhausted_pick'
```

未来 GA4 切片：
- `login_required` → 新用户拉新阻力（改 modal 文案、考虑去掉 sign-in gate）
- `exhausted_pick` → 复购阻力（改价格 / plan 设计）

DB 查 `martinfly1016@gmail.com` 5/11 09:44-10:00 共 7 次 paid 调用（NB2 模型选型测试，与 commit `ccde468` 时间窗匹配）。但 `db-query.ts:95` 的 `testEmails` 只排除 `e2e-test+t{1,2,3}@prompta.jp`，没排除 owner 真实账号。

**Fix (`85d227c`)**: 抽 `_owner_exclusion.ts` 共享清单，加 `martinfly1016@gmail.com` + `yuchao@byte-ad.com`。tool-usage / paywall-hits / `_paywall_users_detail.ts` 统一引用。

### 3. checkout_started=1 但 Stripe 0 成交

GA 显示 5/10 同日 fire 了三个事件：
```

2026-05-10 paywall_purchase_click               hair-color  1
2026-05-10 checkout_started                     hair-color  1
```

`paywall_purchase_click` / `checkout_started` 事件 schema **没有 trigger 字段**（看 `track.ts:119-125`），所以单看 funnel 表无法区分。但按时间戳对照 `docs/session-2026-05-10.md`：

> plumbing test：用 chrome-devtools MCP fire 测试事件...（注：5/13 看日报会有 1 行 trigger=plumbing_test，是测试污染，可忽略）

实际上当时 fire 了 funnel 全部三个事件做端到端测试，**没真的请求 `/api/checkout/hair-color`**（所以 Stripe 没生成 session）。session-2026-05-10.md 漏标了 click + checkout 也是同批污染。

→ **本周真实 funnel**: 20 view（剔除 plumbing 1）→ 0 click → 0 checkout → 0 pay。

## Part B — 用错指标了：copy-rate ≠ 参数化的合适指标

第一次 param 效果分析对照组：

| 组 | sessions | copy-rate |
|---|---|---|
| 参数化 16 slug | 71 | 11.27% |
| 非参数化 top-15 | 76 | 11.84% |

→ 几乎无差异。一度判定为「机制可能没真正提升转化」。

**用户洞察纠错**：copy_prompt 是「最终决策」事件 — 用户调一次 vs 调十次都只 copy 一次（找到满意 prompt 后走人）。**copy-rate 量错了东西**。

正确指标：**用户在页面上的投入度** — bounce / engage / dwell。

### 重做后的数据（2026-05-10 ~ 2026-05-16, 7d）

| 组 | sessions | bounce | engage | dwell |
|---|---|---|---|---|
| 参数化 16 slug | 71 | **32.4%** | **67.6%** | 78.5s |
| 非参数化（所有 prompt 页）| 234 | 41.0% | 59.0% | 389s\* |

**Δ bounce**: -8.6pp 🟢 · **Δ engage**: +8.6pp 🟢

（\*dwell 受少数「开 tab 不关」用户拉偏，bounce / engage 才是干净对比）

调参 slug 单独看更清楚：
- twintail-blonde-sailor-outfit: bounce 16.7% / engage 83.3%
- chibi-girl-walk-29: bounce 20.0% / engage 80.0% / 6.5 events/user
- straight-long-black-hair-glossy: bounce 20.0% / dwell 170.9s / 9.0 events/user

**3 个独立指标交叉验证机制有效**：
| 价值假设 | 量度 | 数据 | 结论 |
|---|---|---|---|
| 用户注意到面板 | 触发率 | 14.1% (10/71 sessions) | ✅ |
| 用户认真在试 | events/调参用户 | 4.5 次 | ✅ |
| 用户更投入 | bounce 差 | -8.6pp | ✅ |

**Plan B (DB 加 paramsJson 列) 暂缓** — 当前 drop-a-file workflow 已经够轻量，痛点不在 "admin 无需 redeploy"。真正的瓶颈是「哪些 slug 该参数化」。
