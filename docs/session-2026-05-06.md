## 1. Mobile LCP fix — 58 URLs out of "needs improvement"

**触发**：用户截图 GSC Core Web Vitals 报告 — 移动端 58 个 URL 「需要改进」（橙色），桌面端 58 个全部「优质」。这是典型的桌面 / 移动表现不对称信号。

**诊断（chrome-devtools MCP perf trace）**：
- LCP 元素 = 首页「最新のプロンプト」第一张卡片图（`era-transform-1970s-classic-before`）
- LCP breakdown：TTFB 20ms / **Resource load delay 561ms** / Load duration 3ms / Render delay 131ms
- 561ms load delay 占 LCP 78.5%，是真凶
- 根因：`<head>` 里塞了 4 个 `<link rel="preload" as="image" fetchpriority="high">`（来自 `priorityCount={4}`）。桌面 4 列布局 4 张都在折叠之上 → 全绿；移动端 1 列布局只有第 1 张可见 → 另外 3 个高优先级 preload 在抢 LCP 图片的 h2 multiplexing window + AVIF 编码 + CDN 边缘

**改动**（commit `aed55e6` `perf(lcp): cut mobile LCP — fewer competing preloads + next/image on detail`）：
1. `src/app/(marketing)/page.tsx` — `priorityCount` 4 → 1（移动端只有第 1 卡需要 preload，desktop 失去微小 preload 收益但带宽充足）
2. `src/app/layout.tsx` — `Noto_Sans_JP` `preload: true → false`（latin-subset 字体在 JP 页面里几乎不参与首屏，preload 等于多塞一条 high-priority 请求和 LCP 抢）
3. `src/app/(marketing)/prompt/[slug]/page.tsx` — 详情页所有原生 `<img>` 换成 `next/image`（之前绕过 next/image 优化器，没有 AVIF / srcset / priority hint）
   - Before/After 样片：`fill` + `aspect-square`（Gemini 输出方图）+ Before 标 `priority`
   - prompt.images：intrinsic `width/height` 来自 DB，第一张在没有 sample 时标 `priority`
4. `src/lib/data/index.ts` — `NormalizedPrompt.images` 类型加 `width/height`（DB PromptImage 已经有这两列，只是 normalize 函数没透出）

**预期**：
- 移动端 LCP load delay 从 ~561ms → ~150-200ms
- photo-edit 详情页（最近上线，58 里大头）原本 Before 图被 `loading="lazy"` 反而成了 LCP 元素，被浏览器延后 → 现在 priority + AVIF 应有 1.5-2s 提升

**验证路径**：GSC 字段数据 24-72h 后更新，看「移动设备」橙线是否回落到绿区。

## 2. English i18n — Step 1：`/en/tools/personal-color-analysis`

**触发**：用户提出 Personal Color tool 本身衍生自英文搜索词，应做英文版 SEO。讨论后定路线：subdirectory（不开子域名），先只做这一个工具页验证 4-6 周。

### 路线决策

| 选项 | 评估 |
|---|---|
| ✅ `/en/tools/personal-color-analysis` 子目录 + hreflang | 选 — SEO 信号集中，不需新基础设施 |
| ❌ `en.prompta.jp` 子域名 | 弃 — 域名权重需重新积累 |
| ❌ 独立 `.com` | 弃 — 暂时成本太高，等流量验证 |

**主要 trade-off**：`.jp` ccTLD 是 EN 排名负面信号（Google 默认绑日本市场），唯一对策是 hreflang + JSON-LD `inLanguage:en` + 反向链接。注：GSC 旧的「International Targeting / Country」菜单 Google 在 2024 年下半年已移除，新版界面没有手动 geo-target 设置可调。

### 本次实现（commit `b01a463` + `e73d0d1`）

**新页面**：`src/app/(marketing)/en/tools/personal-color-analysis/page.tsx`
- 全英文 hero + 「What is personal color analysis?」长尾段
- 6 题 EN FAQ，瞄准：`personal color analysis` (110K mo) / `what season am i` (50K mo) / `ai color analysis` / `find my season`
- Sample mock result 用英文调色板名（Powder Blue / Mauve / Ash Blonde / Soft Coral …）
- FAQPage + WebApplication JSON-LD（`inLanguage: 'en'`，含 aggregateRating 4.7/120 用于 SERP rich snippet）
- 内嵌 `<PersonalColorQuotaGate locale="en" />`

**Gate 组件 i18n**：`src/components/tools/PersonalColorQuotaGate.tsx`
- 加 `locale?: 'ja' | 'en'` prop，默认 'ja' 向后兼容
- 所有用户可见字符串（status chip / errors / modal / recovery / analyzing overlay / result scaffolding）外置到 `STRINGS` 表
- EN locale 在 result 卡片上显示一个黄色提示框：「AI-generated descriptions and color names are currently in Japanese. Full English output rolls out in next update.」（Gemini 输出仍 JP，下一步再改）

**Hreflang 双向**：
- JP 页 metadata.alternates.languages 加 `en` + `x-default → ja`
- EN 页 metadata.alternates.languages 同上
- Sitemap 两条 URL 都带 `alternates.languages`，Search Console 当成 variants 不当 duplicate

**Stripe checkout locale 路由**：
- `POST /api/checkout/personal-color?locale=en` → Stripe Checkout `locale: 'en'` + `success_url` / `cancel_url` 回跳到 `/en/tools/...`

**可见的语言切换链接**：
- EN 页 hero 有 "🇯🇵 日本語版" pill（最初就有）
- JP 页 hero 加 "🇺🇸 English version" pill（commit `e73d0d1`）— Google 对页面内可见 anchor 的语言信号权重高于 hreflang alone

## 关键技术决策记录

1. **mobile-first `priorityCount=1`**：桌面虽然 4 列首屏 4 卡可见，但 desktop 已经全绿，宁愿牺牲微小 desktop preload 收益换取移动端 LCP；preload 不是 image loading 的必要条件，普通 eager loading 已足够
